# Blog Writing Assistant

用于把用户的素材整理成当前 Docusaurus 站点里的笔记或博客。

这个 skill 的目标不是替用户写一篇“万能 AI 文”，而是把用户已有的想法、草稿、提问、工作记录整理成适合个人站点发布的内容。

## 适用场景

- 用户说“帮我写一篇博客/笔记”。
- 用户给出一段工作总结、问题清单、会议记录、草稿，希望整理进站点。
- 用户提供一个本地 Markdown 文件，希望迁移成博客或笔记。
- 用户只有一些线索，希望 AI 追问后生成一篇文章。
- 用户要求“按我之前那篇博客/笔记的风格写”。

## 必问信息

除非用户已经明确提供，否则先询问以下三类信息。一次性问完，不要反复打断。

### 1. 内容类型

让用户选择：

- **写博客**：放入 `blog/`，适合完整文章、对外表达、阅读路线、经验分享。
- **写笔记**：放入 `docs/`，适合工作思考、问题拆解、技术记录、项目沉淀。

如果用户不确定，优先建议：

- 有明确观点、叙事和读者对象：博客。
- 偏记录、拆问题、后续继续补充：笔记。

### 2. 来源模式

让用户选择：

- **迁移已有内容**：用户已有 Markdown、草稿、链接或长文本。目标是整理、补 front matter、调整排版，尽量不改变原意。
- **根据线索生成**：用户只有关键词、问题清单、结论、口述想法。目标是先追问必要信息，再生成结构完整的文章。

如果是迁移已有内容，必须确认源文件路径或完整文本。

如果是根据线索生成，至少追问：

- 这篇内容想解决什么问题？
- 读者是谁？
- 有没有必须保留的原话、观点或例子？
- 是否需要偏工作记录、经验总结、教程，还是观点文章？

### 3. 写作风格

把下面两个当前站内风格加入选项列表，并允许用户自定义。

推荐选项：

- **个人博客式学习路线**：参考 `blog/2026-05-08-trellis-harness-learning-guide.mdx`。特点是第一人称或轻叙事开头，语气自然，不端着；围绕一个技术对象给读者一条可执行阅读路线；包含心智模型、推荐顺序、动手路径和常见误区。适合博客。
- **问题驱动技术笔记**：参考 `docs/tech-notes/harness-workflow-automation.mdx`。特点是从一个工作中遇到的真实问题切入，先定义核心矛盾，再拆成几个追问；用分层结构分析，不急着给绝对答案；最后沉淀成后续可以继续追的问题。适合笔记。
- **实操教程型**：目标明确，步骤清楚，适合“怎么做”。
- **复盘总结型**：按背景、过程、问题、决策、结果、经验组织，适合项目复盘。
- **观点短文型**：围绕一个判断展开，篇幅更短，语气更鲜明。
- **清单备忘型**：以结构化列表为主，适合快速查阅和后续补充。

风格不是格式模板。选定风格后，要保留用户自己的表达，不要把所有文章写成同一种声音。

## 当前两个站内风格说明

### 个人博客式学习路线

对应文章：`blog/2026-05-08-trellis-harness-learning-guide.mdx`。

写作特征：

- 标题像个人博客，不像正式文档，例如“我是怎么读懂 Trellis Harness 的”。
- 开头先说读者会遇到的真实困惑。
- 中间用“先建立心智模型”“推荐学习顺序”“hands-on 路线”“常见误区”组织内容。
- 技术名词可以多，但要给读者一条能照着走的路线。
- 适合把一件复杂技术讲成“我会怎么读、怎么学、怎么验证”。

常用结构：

```text
开头：为什么这件事容易让人困惑
  -> 心智模型
  -> 推荐学习顺序
  -> 动手探索路线
  -> 常见误区
  -> 总结
```

### 问题驱动技术笔记

对应笔记：`docs/tech-notes/harness-workflow-automation.mdx`。

写作特征：

- 标题直接但不僵硬，例如“Harness 工作流怎么快速搭起来”。
- 开头从“最近在想一个问题”这类工作思考切入。
- 先写完整链路或背景，再指出真正卡住的核心问题。
- 每个小节回答一个追问，不急着包装成结论。
- 适合放入 docs，作为持续演进的技术思考。

常用结构：

```text
当前观察
  -> 完整链路
  -> 核心矛盾
  -> 几个关键追问
  -> 分层分析
  -> 理想流程
  -> 后续问题
```

## 生成规则

### 写博客

博客文件放在 `blog/`。

文件名格式：

```text
YYYY-MM-DD-short-slug.mdx
```

front matter 至少包含：

```mdx
---
slug: short-slug
title: 标题
description: 一句话描述
tags: [Tag1, Tag2]
---
```

正文开头应该在 2 到 4 段内说明问题背景，并在合适位置加入：

```mdx
{/* truncate */}
```

博客标题要像个人文章，避免过度正式、过度长、过度像论文。

### 写笔记

笔记文件优先放在：

- `docs/tech-notes/`：技术概念、工具、流程、方法论。
- `docs/project-notes/`：当前博客站点或具体项目改造记录。
- `docs/troubleshooting/`：排障、问题定位、解决记录。

front matter 至少包含：

```mdx
---
sidebar_position: 数字
title: 标题
---
```

正文使用一级标题重复标题：

```mdx
# 标题
```

笔记可以保留问题感，不必写成完全封闭的结论。

## 迁移已有内容流程

1. 读取源内容。
2. 判断更适合博客还是笔记；如果用户已指定，以用户为准。
3. 保留用户核心观点、原始问题、关键例子。
4. 修正错别字、口语断裂、编号混乱和 Markdown 结构问题。
5. 补充必要的标题、front matter、摘要、tags 或 sidebar_position。
6. 不要擅自删掉用户明确表达的判断；如果需要弱化，保留意思但调整语气。
7. 创建或更新目标文件。
8. 运行 `npm run build` 验证 MDX。

## 根据线索生成流程

1. 先问清内容类型、来源模式和风格。
2. 根据线索追问最多 3 到 5 个关键问题；如果线索已经足够，不要继续问。
3. 先组织文章骨架，再写正文。
4. 保持用户的表达重心，不要为了完整而引入太多用户没提过的新观点。
5. 如果需要补充判断，用“可能的方向”“可以继续追的问题”等方式表达，不要伪装成既定事实。
6. 创建目标文件后运行 `npm run build`。

## 质量标准

- 标题自然，像人写的，不像自动生成的报告标题。
- 开头能说明“为什么要读这篇”。
- 每个小节只解决一个问题。
- 列表要服务于表达，不要堆砌。
- 技术笔记可以保留未解决问题，但要把问题拆清楚。
- 博客文章要有阅读路线或观点推进，不只是素材拼贴。
- 中文和英文技术词混用时保持自然，不要强行翻译固定术语。
- 最后必须确保 Docusaurus 构建通过。

## 交付格式

完成后告诉用户：

- 新增或修改了哪个文件。
- 使用了哪种内容类型和风格。
- 构建是否通过。
- 如果本地站点已启动，可以给出页面路径。
