很多 CSS 性能建议听起来像经验口诀：少用阴影、动画用 transform、避免频繁重排。

但如果把浏览器渲染过程拆开看，这些建议背后的原因会清晰很多。

## 浏览器渲染的大致流程

一个页面从 HTML 和 CSS 到屏幕上可见，大致会经历这些阶段：

```txt
DOM + CSSOM
  -> Render Tree
  -> Layout
  -> Paint
  -> Composite
```

不同 CSS 属性会触发不同阶段的工作。

比如改变 `width`，浏览器需要重新计算布局；改变 `background-color`，通常只需要重新绘制；改变 `transform`，很多情况下可以只走合成阶段。

## Layout：最容易牵连其他元素

布局阶段会计算每个元素的位置和尺寸。

下面这些属性通常会影响布局：

- `width`
- `height`
- `margin`
- `padding`
- `border-width`
- `display`
- `position`
- `top` / `left` / `right` / `bottom`

如果一个元素的尺寸变化会影响兄弟节点或父节点，那么浏览器就可能重新计算一大片区域。

例如用 `left` 做动画：

```css
.panel {
  position: relative;
  left: 0;
  transition: left 200ms ease;
}

.panel.open {
  left: 240px;
}
```

这可能触发布局计算。更推荐用 `transform`：

```css
.panel {
  transform: translateX(0);
  transition: transform 200ms ease;
}

.panel.open {
  transform: translateX(240px);
}
```

## Paint：视觉效果也有成本

绘制阶段负责把边框、背景、文字、阴影等视觉内容画出来。

下面这些属性通常更容易影响绘制：

- `color`
- `background`
- `box-shadow`
- `border-radius`
- `filter`

复杂阴影和滤镜在大面积元素上会比较贵，尤其是当它们频繁变化时。

这不是说不能用，而是要避免在滚动区域或大尺寸元素上做持续动画。

## Composite：动画的友好路径

合成阶段会把不同图层组合到一起。

如果动画只影响 `transform` 或 `opacity`，浏览器很多时候可以跳过布局和绘制，直接在合成线程处理。

这就是为什么常见建议是：

```css
.item {
  transition:
    transform 160ms ease,
    opacity 160ms ease;
}
```

但也不要滥用 `will-change`：

```css
.card {
  will-change: transform;
}
```

它会提示浏览器提前做优化，但也可能带来额外内存成本。适合短时间即将发生动画的元素，不适合全站到处写。

## containment：告诉浏览器边界在哪里

现代 CSS 里有一个很实用的性能工具：`contain`。

```css
.widget {
  contain: layout paint;
}
```

它可以告诉浏览器：这个组件内部的布局和绘制影响尽量限制在自身范围内。

对于独立卡片、侧边栏小组件、复杂列表项，这会让浏览器更容易缩小重新计算的范围。

## 实用检查清单

排查 CSS 性能问题时，可以先看这些点：

- 动画是否在改 `width`、`height`、`left`、`top`
- 大面积元素是否有复杂阴影、滤镜或模糊
- 滚动容器里是否有太多动态视觉效果
- 是否可以用 `transform` 替代位置变化
- 是否可以用 `contain` 限制复杂组件影响范围

## 小结

CSS 性能不是记住一堆禁忌，而是理解每个属性会让浏览器重新做什么工作。

如果一条动画可以只走合成阶段，它通常会比频繁触发布局和绘制更稳定。
